Node中的模块系统之exports和module.exports的关系

1.什么是文件模块化

  • 文件作用域
  • 通信规则
    • 加载
    • 导出

2.CommonJS模块规则

在Node中的JavaScript有一个很重要的概念——模块系统。

  • 模块作用域
  • 使用require方法加载模块
  • 使用exports接口对象导出模块中的成员

2.1 加载require

语法：

var 自定义变量名 = require('模块')

作用：

  • 执行被加载模块中的代码
  • 得到被加载模块中的exports导出接口对象

2.2 导出exports

  • Node中时模块作用域，默认文件中的左右成员只在当前文件模块内部有效
  • 对希望可以被其他模块访问的成员，需要把这些公开的成员挂载到exports接口对象中

导出多个成员（在对象中导出）：

exports.name = 'eagle';
exports.age = 18;
exports.sayHi = function () &#123;
    console.log('Hi~');
&#125;
exports.family = &#123;
    father: 'father';
    morther: 'morther';
&#125;

导出单个成员（直接拿到的函数、字符串等，不使用导出exports对象）：

module.exports = 'hello';

由于是单个赋值，将内容直接赋值给了exports，所以以下情况会导致覆盖：

// 导出第一个
module.exports = 'i am the first exports';

// 导出第二个
module.exports = function () &#123;
    console.log('i am the second exports');
&#125;
// 该情况只会导出第二个，后者会覆盖前者

也可以通过以对象形式一次性直接导出多个成员：

module.exports = &#123;
    name: 'eagle',
    age: 18,
    sayHi: function () &#123;
        console.log('Hi~');
    &#125;,
    family: &#123;
        father: 'father';
        morther: 'morther';
    &#125;
&#125;

3.模块原理

注意：exports是module.exports的一个引用！！！

console.log(exports === module.exports) // true

exports.foo = 'bar';
// 等价于
module.exports.foo = 'bar'

在Node中，每一个模块的内部都有一个自己的module对象，而该对象中，有一个成员叫exports：

var module = &#123;
    exports: &#123;
        // 实际添加的模块内容
    &#125;,
    ...
&#125;

而在代码的最后，默认有一个return module.exports，谁require了这个模块，谁就会获得这个module.exports接口对象

而实际写入文件的模块内容，其实就是在修改module对象的成员exports对象中的内容

而exports是对module.exports对象的一个引用，也就是说

所以：

// foo.js文件
exports.a = 'foo';
module.exports.b = 'bar'; 

// 实际上：
// var module = &#123;
//        exports: &#123;
//     &#125;
// &#125;
// var exports = module.exports; // exports 引用 module.exports
// 
// exports.a = 'foo';
// module.exports.b = 'bar';
//
// return module.exports;

而当我们在另一个文件require时：

// bar.js文件
var foo = require('./foo'); // .js后缀可以省略
console.log(foo);
// 打印内容为 &#123; a: 'foo', b: 'bar' &#125;

怼怼1号：如此说来，那 exports 和module.exports岂不是没什么区别，那后者那么长，谁还会挑长的写呢？

这确实是一个好问题，如果二者没有区别，那么他们必然没有存在的意义，那么区别在哪里呢？

仔细看前文的模块规则，发现只有在分次导出多个成员时，才存在exports和module.exports都合理的情况，而后面的导出单个成员用例，以及一次性导出多个成员用例中，却没有出现直接使用exports的情况

究其原因，还是那句话，exports只是对module.exports的一个引用，当对module对象内的exports对象进行修改时，二者没有区别，甚至使用exports更好（为什么？因为短！）

因为var exports = module.exports;的存在，他们指向的是同一个地址，但是归根究底，他俩始终不是同一个东西，当直接赋值时，如果使用exports = 'foo'的方式，那么原本exports与module.exports之间的引用关系就会断开，此时他们指向的地址不同，exports指向'foo'而不再是module.exports

而前文提到，代码最后总是默认return module.exports，返回的不是exports，所以该方法无法实现导出。

Talk is cheap, show me the code!

先举一个简单的例子

var foo = &#123;&#125;
var bar = foo; // 此时 bar 与 foo 建立了引用关系，他们所指向的为同一个空对象

foo.name = 'eagle';
bar.age = 18;

console.log(foo); // &#123; name: 'eagle', age: 18 &#125;
console.log(bar); // &#123; name: 'eagle', age: 18 &#125;

bar.name = 'crow';

console.log(foo); // 打印内容为 &#123; name: 'crow', age: 18 &#125;

// 但是当我们对例子中的 bar 进行重新赋值操作时, bar 与 foo之间的引用关系断开
bar = &#123;&#125;;

console.log(foo); // &#123; name: 'crow', age: 18 &#125;
console.log(bar); // &#123;&#125;

// 所以此时对 bar 做任何操作，都不会影响到 foo 的内容
bar.name = 'ealge';
bar.age = 16;

console.log(foo); // &#123; name: 'crow', age: 18 &#125;
console.log(bar); // &#123; name: 'ealge', age: 16 &#125;

// return foo;

这个例子很好理解，那么同理，我们来梳理一下exports和module.exports的关系

// module.exports => foo
// exports => bar

// var module = &#123;
//     exports: &#123;
//     &#125;
// &#125;
// var exports = module.exports;
// 以上为默认代码

// 此时我们对模块内容进行修改
module.exports.name = 'eagle';
exports.age = 18;

// 因为返回的是module.exports，所以我们也只打印他，模拟被require情形
console.log(module.exports); // &#123; name: 'eagle', age: 18 &#125;

exports.name = 'crow';
console.log(module.exports); // &#123; name: 'crow', age: 18 &#125;

// 以上为分次导出多个对象，现在我们想进行直接赋值导出如果使用 exports
exports = 'abc'; // 此时 exports 与 module.exports 之间的引用关系断开，但返回的仍然时module.exports，无论exports如何修改都无法干预到module.exports
console.log(module.exports); // &#123; name: 'crow', age: 18 &#125;

// 同理一次导出多个成员也一样，由于对exports进行了重新赋值，断开了引用关系，所以无法通过exports修改module.exports
exports = &#123;
    name: 'abc',
    age: 3
&#125;
console.log(module.exports); // &#123; name: 'crow', age: 18 &#125;

// 以下为默认代码
// return module.exports;

怼怼2号：那这样的话，我们在后面对其进行重新建立引用关系不就行了吗？

有道理啊，重新建立关系，就能够对模块内容进行修改了不是吗？

答案是肯定的

// var module = &#123;
//     exports: &#123;
//     &#125;
// &#125;
// var exports = module.exports;
// 以上为默认代码

// 此时我们对模块内容进行修改
module.exports.name = 'eagle';
exports.age = 18;

// 老规矩，因为返回的是module.exports，所以我们也只打印他，模拟被require情形
console.log(module.exports); // &#123; name: 'eagle', age: 18 &#125;

// 我们先'断开链接'，由于引用关系断开，所以我们对exports的修改无法影响到返回内容
exports = &#123;&#125;;
exports.name = 'abc';
exports.age = 3;
console.log(module.exports); // &#123; name: 'eagle', age: 18 &#125;

// 现在我们重新建立引用关系，此时因为对 exports 重新赋值，所以上面的修改内容被丢弃，exports 指向 module.exports
exports = module.exports;

// 修改模块内容
exports.name = 'crow';
exports.age = 16;
// 成功修改模块内容
console.log(module.exports); // &#123; name: 'crow', age: 16 &#125;

// 以下为默认代码
// return module.exports;

综上所述，重新建立连接确实可以实现继续对模块内容进行修改，但是重新建立连接以后exports依然还是只能分次导出多个成员，无法取代module.exports的地位。

此时有一种新的情况出现，值得思考

// var module = &#123;
//     exports: &#123;
//     &#125;
// &#125;
// var exports = module.exports;
// 以上为默认代码

// 此时我们对模块内容进行修改
module.exports.name = 'eagle';
exports.age = 18;
console.log(module.exports); // &#123; name: 'eagle', age: 18 &#125;

// 当我们用赋值方法一次性导出多个成员时
module.exports = &#123;
    name: 'crow',
    age: 16,
    sayHi: function () &#123;
        console.log('Hi~');
    &#125;
&#125;

console.log(module.exports); // &#123; name: 'crow', age: 16, sayHi: [Function: sayHi] &#125;

// 但此时我们如果在想用exports进行修改内容
exports.name = 'abc';
exports.age = 3;

console.log(module.exports); // &#123; name: 'crow', age: 16, sayHi: [Function: sayHi] &#125; 
// 显而易见，修改失败
// 为什么呢？前面不是写了 exports = module.exports 了嘛？
// 我们来打印一下
console.log(exports === module.exports); // false

// 其实与 exports 重新赋值概念相同，当我们使用 module.exports 重新赋值导出时，并不是将原来地址内容进行覆盖修改，而是重新开辟了新的地址来储存新的对象，而原来的地址内容是由 exports 指向

console.log(exports); // &#123; name: 'eagle', age: 18 &#125;

// 以下为默认代码
// return module.exports;

最后引用一下Node官方文档的内容：

exports允许使用快捷方式，因此module.exports.f = ...可以更简洁地编写为exports.f = ...。但是，请注意，就像任何变量一样，如果将新值分配给exports，则它不再绑定到module.exports。

当该module.exports属性完全被新对象替换时，通常也要重新分配exports。

总结一下：emm…..如果你实在是搞不懂exports和module.exports之间的区别，那就忘记exports，只使用module.exports也是没有问题的，over！
